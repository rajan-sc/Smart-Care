## 1. Design Philosophy

### Normalization

All tables are designed to **3NF (Third Normal Form)** to eliminate data redundancy and prevent update anomalies. Denormalization is introduced only at the caching/read-model layer (Redis) — never in the source-of-truth schema.

- Vitals use a **type discriminator** column (`vital_type`) rather than separate columns for each measurement, allowing the schema to scale to new vital types without DDL changes.
- Medicine timings are stored as `JSONB` arrays — a pragmatic deviation from strict normalization, since timings are always read/written as a unit and never queried individually in joins.

### Referential Integrity

Every foreign key is **explicitly declared** with a defined `ON DELETE` behavior:

| Scenario | ON DELETE Policy |
|---|---|
| Parent is a user owning child records | `CASCADE` — when a user is hard-deleted (admin action), all owned records go with them |
| Reference to a lookup/contextual entity | `SET NULL` — preserve the child record even if the referenced entity is removed |
| Link/junction tables | `CASCADE` — the link has no meaning without both sides |

### Soft Deletes

All **business entities** carry a `deleted_at TIMESTAMPTZ DEFAULT NULL` column. A row with `deleted_at IS NOT NULL` is considered logically deleted. Hard deletes are reserved for:

- GDPR/data-erasure requests (admin-only operation)
- Expired refresh tokens (automated cleanup)

### Primary Keys

All tables use `UUID v4` primary keys generated by PostgreSQL's `gen_random_uuid()`. This provides:

- No sequential ID enumeration attacks
- Safe distributed ID generation (no coordination needed)
- Compatibility with Prisma's `@default(uuid())` directive

## 3. Enum Definitions

All enums are defined as PostgreSQL custom types. Prisma maps these natively.

```sql
-- User role within the platform
CREATE TYPE user_role AS ENUM (
  'patient',
  'doctor',
  'caregiver',
  'admin'
);

-- Biological gender (for medical context)
CREATE TYPE user_gender AS ENUM (
  'male',
  'female',
  'other',
  'prefer_not_to_say'
);

-- Appointment lifecycle
CREATE TYPE appointment_status AS ENUM (
  'booked',        -- patient has reserved the slot
  'confirmed',     -- doctor/system confirmed
  'in_progress',   -- patient is currently being seen
  'completed',     -- visit finished
  'cancelled',     -- cancelled by patient or doctor
  'no_show'        -- patient did not arrive
);

-- Live queue token state
CREATE TYPE queue_token_status AS ENUM (
  'waiting',       -- in the queue, not yet called
  'called',        -- doctor has called this token
  'in_progress',   -- patient is with the doctor
  'completed',     -- consultation done
  'skipped'        -- patient was absent when called
);

-- Medicine repeat frequency
CREATE TYPE repeat_type AS ENUM (
  'daily',
  'weekly',
  'monthly',
  'as_needed',
  'custom'         -- uses JSONB field for custom schedule
);

-- Medication log outcome
CREATE TYPE medication_log_status AS ENUM (
  'pending',       -- scheduled, not yet acted upon
  'taken',         -- patient confirmed intake
  'skipped',       -- patient explicitly skipped
  'missed',        -- window elapsed with no action
  'snoozed'        -- patient deferred, will be reminded again
);

-- Vital measurement category
CREATE TYPE vital_type AS ENUM (
  'blood_pressure',
  'glucose',
  'pulse',
  'weight',
  'temperature',
  'spo2'
);

-- Notification category
CREATE TYPE notification_type AS ENUM (
  'medication_reminder',
  'appointment_reminder',
  'queue_update',
  'vital_alert',
  'caregiver_alert',
  'system'
);

-- Notification delivery state
CREATE TYPE notification_status AS ENUM (
  'pending',       -- created, not yet delivered
  'sent',          -- pushed to client (Socket.IO / push)
  'read',          -- user opened/acknowledged
  'dismissed'      -- user dismissed without reading
);

-- Caregiver access level
CREATE TYPE caregiver_permission AS ENUM (
  'view_only',     -- can view vitals, logs, appointments
  'manage'         -- can also book appointments, log meds
);

-- Caregiver link state
CREATE TYPE link_status AS ENUM (
  'pending',       -- invitation sent, not yet accepted
  'active',        -- link is live
  'revoked'        -- patient or caregiver revoked access
);

-- Audit log action verbs
CREATE TYPE audit_action AS ENUM (
  'create',
  'update',
  'delete',
  'login',
  'logout',
  'password_change',
  'role_change',
  'verify_email'
);
```

## 4. Table Definitions

### 4.1 `users`

The central identity table. Every person in the system has exactly one user record.

```sql
CREATE TABLE users (
  id             UUID          PRIMARY KEY DEFAULT gen_random_uuid(),
  email          VARCHAR(255)  NOT NULL,
  password_hash  TEXT          NOT NULL,
  role           user_role     NOT NULL DEFAULT 'patient',
  is_verified    BOOLEAN       NOT NULL DEFAULT FALSE,
  first_name     VARCHAR(100)  NOT NULL,
  last_name      VARCHAR(100)  NOT NULL,
  phone          VARCHAR(20),
  date_of_birth  DATE,
  gender         user_gender,
  avatar_url     TEXT,
  created_at     TIMESTAMPTZ   NOT NULL DEFAULT now(),
  updated_at     TIMESTAMPTZ   NOT NULL DEFAULT now(),
  deleted_at     TIMESTAMPTZ
);
```

**Constraints:**

| Type | Name | Definition | Rationale |
|---|---|---|---|
| UNIQUE | `uq_users_email` | `email WHERE deleted_at IS NULL` | Allows re-registration after hard delete; prevents duplicate active accounts |
| CHECK | `ck_users_email_format` | `email ~* '^[A-Za-z0-9._%+-]+@[A-Za-z0-9.-]+\.[A-Za-z]{2,}$'` | Basic email format validation at DB level |
| CHECK | `ck_users_phone_format` | `phone IS NULL OR phone ~ '^\+?[0-9]{7,15}$'` | E.164-like phone validation |

**Indexes:**

```sql
-- Unique partial index: only active (non-deleted) users must have unique emails
CREATE UNIQUE INDEX idx_users_email_active
  ON users (email)
  WHERE deleted_at IS NULL;

-- For login lookups
CREATE INDEX idx_users_email_role
  ON users (email, role)
  WHERE deleted_at IS NULL;

-- For admin user listing sorted by creation
CREATE INDEX idx_users_created_at
  ON users (created_at DESC);

-- For filtering by role
CREATE INDEX idx_users_role
  ON users (role)
  WHERE deleted_at IS NULL;
```

### 4.2 `doctor_profiles`

Extended profile for users with `role = 'doctor'`. 1:1 relationship with `users`.

```sql
CREATE TABLE doctor_profiles (
  id                    UUID          PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id               UUID          NOT NULL,
  specialization        VARCHAR(100)  NOT NULL,
  clinic_name           VARCHAR(200),
  clinic_address        TEXT,
  availability          JSONB         NOT NULL DEFAULT '[]',
  max_patients_per_slot INTEGER       NOT NULL DEFAULT 10,
  consultation_fee      DECIMAL(10,2),
  is_accepting          BOOLEAN       NOT NULL DEFAULT TRUE,
  created_at            TIMESTAMPTZ   NOT NULL DEFAULT now(),
  updated_at            TIMESTAMPTZ   NOT NULL DEFAULT now(),

  CONSTRAINT fk_doctor_profiles_user
    FOREIGN KEY (user_id) REFERENCES users(id) ON DELETE CASCADE,
  CONSTRAINT uq_doctor_profiles_user_id
    UNIQUE (user_id),
  CONSTRAINT ck_doctor_max_patients
    CHECK (max_patients_per_slot > 0 AND max_patients_per_slot <= 100),
  CONSTRAINT ck_doctor_consultation_fee
    CHECK (consultation_fee IS NULL OR consultation_fee >= 0)
);
```

**`availability` JSONB Structure:**

```jsonc
[
  {
    "day": "monday",          // lowercase day name
    "slots": [
      { "start": "09:00", "end": "12:00" },
      { "start": "14:00", "end": "17:00" }
    ]
  }
]
```

**Indexes:**

```sql
-- Lookup by user_id (already unique, covered by constraint)
-- Search doctors by specialization
CREATE INDEX idx_doctor_profiles_specialization
  ON doctor_profiles (specialization)
  WHERE is_accepting = TRUE;

-- GIN index for querying availability JSONB
CREATE INDEX idx_doctor_profiles_availability
  ON doctor_profiles USING GIN (availability);
```

### 4.3 `appointments`

Booking records linking patients to doctors at specific time slots.

```sql
CREATE TABLE appointments (
  id            UUID                PRIMARY KEY DEFAULT gen_random_uuid(),
  patient_id    UUID                NOT NULL,
  doctor_id     UUID                NOT NULL,
  slot_start    TIMESTAMPTZ         NOT NULL,
  slot_end      TIMESTAMPTZ         NOT NULL,
  token_number  INTEGER,
  status        appointment_status  NOT NULL DEFAULT 'booked',
  reason        TEXT,
  notes         TEXT,
  created_at    TIMESTAMPTZ         NOT NULL DEFAULT now(),
  updated_at    TIMESTAMPTZ         NOT NULL DEFAULT now(),
  deleted_at    TIMESTAMPTZ,

  CONSTRAINT fk_appointments_patient
    FOREIGN KEY (patient_id) REFERENCES users(id) ON DELETE CASCADE,
  CONSTRAINT fk_appointments_doctor
    FOREIGN KEY (doctor_id) REFERENCES users(id) ON DELETE CASCADE,
  CONSTRAINT ck_appointments_slot_order
    CHECK (slot_end > slot_start),
  CONSTRAINT ck_appointments_token_positive
    CHECK (token_number IS NULL OR token_number > 0)
);
```

**Constraints:**

| Type | Name | Definition | Rationale |
|---|---|---|---|
| CHECK | `ck_appointments_slot_order` | `slot_end > slot_start` | Prevents zero-length or inverted slots |
| CHECK | `ck_appointments_token_positive` | `token_number > 0` | Token numbers are natural numbers |
| UNIQUE | `uq_appointments_doctor_token` | `(doctor_id, DATE(slot_start), token_number)` | No duplicate tokens per doctor per day |

**Indexes:**

```sql
-- Unique token per doctor per day (partial — only non-deleted, non-cancelled)
CREATE UNIQUE INDEX idx_appointments_doctor_day_token
  ON appointments (doctor_id, (slot_start::DATE), token_number)
  WHERE deleted_at IS NULL AND status NOT IN ('cancelled', 'no_show');

-- Patient appointment history
CREATE INDEX idx_appointments_patient_slot
  ON appointments (patient_id, slot_start DESC)
  WHERE deleted_at IS NULL;

-- Doctor schedule lookup
CREATE INDEX idx_appointments_doctor_slot
  ON appointments (doctor_id, slot_start)
  WHERE deleted_at IS NULL;

-- Filter by status
CREATE INDEX idx_appointments_status
  ON appointments (status)
  WHERE deleted_at IS NULL;

-- Date-range queries on slot_start (frequently used)
CREATE INDEX idx_appointments_slot_start
  ON appointments (slot_start);
```

### 4.4 `queue_tokens`

Real-time queue management for live clinic visits. Each token belongs to one appointment.

```sql
CREATE TABLE queue_tokens (
  id              UUID                PRIMARY KEY DEFAULT gen_random_uuid(),
  appointment_id  UUID                NOT NULL,
  doctor_id       UUID                NOT NULL,
  token_number    INTEGER             NOT NULL,
  status          queue_token_status  NOT NULL DEFAULT 'waiting',
  called_at       TIMESTAMPTZ,
  completed_at    TIMESTAMPTZ,
  queue_date      DATE                NOT NULL DEFAULT CURRENT_DATE,
  created_at      TIMESTAMPTZ         NOT NULL DEFAULT now(),
  updated_at      TIMESTAMPTZ         NOT NULL DEFAULT now(),

  CONSTRAINT fk_queue_tokens_appointment
    FOREIGN KEY (appointment_id) REFERENCES appointments(id) ON DELETE CASCADE,
  CONSTRAINT fk_queue_tokens_doctor
    FOREIGN KEY (doctor_id) REFERENCES users(id) ON DELETE CASCADE,
  CONSTRAINT uq_queue_tokens_appointment
    UNIQUE (appointment_id),
  CONSTRAINT ck_queue_tokens_number
    CHECK (token_number > 0)
);
```

**Indexes:**

```sql
-- Live queue: doctor's tokens for today, ordered by token number
CREATE UNIQUE INDEX idx_queue_tokens_doctor_date_number
  ON queue_tokens (doctor_id, queue_date, token_number);

-- Current queue state for a doctor
CREATE INDEX idx_queue_tokens_doctor_status
  ON queue_tokens (doctor_id, status)
  WHERE status IN ('waiting', 'called', 'in_progress');
```

### 4.5 `medicines`

Medication prescriptions/schedules for a user. A user may have many active medicines.

```sql
CREATE TABLE medicines (
  id             UUID         PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id        UUID         NOT NULL,
  medicine_name  VARCHAR(200) NOT NULL,
  dosage         VARCHAR(100) NOT NULL,
  timings        JSONB        NOT NULL DEFAULT '[]',
  repeat_type    repeat_type  NOT NULL DEFAULT 'daily',
  start_date     DATE         NOT NULL DEFAULT CURRENT_DATE,
  end_date       DATE,
  is_active      BOOLEAN      NOT NULL DEFAULT TRUE,
  created_at     TIMESTAMPTZ  NOT NULL DEFAULT now(),
  updated_at     TIMESTAMPTZ  NOT NULL DEFAULT now(),
  deleted_at     TIMESTAMPTZ,

  CONSTRAINT fk_medicines_user
    FOREIGN KEY (user_id) REFERENCES users(id) ON DELETE CASCADE,
  CONSTRAINT ck_medicines_date_order
    CHECK (end_date IS NULL OR end_date >= start_date)
);
```

**`timings` JSONB Structure:**

```jsonc
["08:00", "14:00", "21:00"]   // 24-hour HH:MM strings
```

**Indexes:**

```sql
-- Active medicines for a user (most common query)
CREATE INDEX idx_medicines_user_active
  ON medicines (user_id)
  WHERE is_active = TRUE AND deleted_at IS NULL;

-- For medication reminder cron: find all active medicines
CREATE INDEX idx_medicines_active_repeat
  ON medicines (repeat_type, is_active)
  WHERE is_active = TRUE AND deleted_at IS NULL;

-- GIN index for querying specific timing values
CREATE INDEX idx_medicines_timings
  ON medicines USING GIN (timings);
```

### 4.6 `medication_logs`

Immutable audit trail of each scheduled medication event and its outcome.

```sql
CREATE TABLE medication_logs (
  id              UUID                   PRIMARY KEY DEFAULT gen_random_uuid(),
  medicine_id     UUID                   NOT NULL,
  user_id         UUID                   NOT NULL,
  scheduled_time  TIMESTAMPTZ            NOT NULL,
  taken_at        TIMESTAMPTZ,
  status          medication_log_status  NOT NULL DEFAULT 'pending',
  skip_reason     TEXT,
  created_at      TIMESTAMPTZ            NOT NULL DEFAULT now(),

  CONSTRAINT fk_medication_logs_medicine
    FOREIGN KEY (medicine_id) REFERENCES medicines(id) ON DELETE CASCADE,
  CONSTRAINT fk_medication_logs_user
    FOREIGN KEY (user_id) REFERENCES users(id) ON DELETE CASCADE,
  CONSTRAINT ck_medication_logs_taken_at
    CHECK (
      (status = 'taken' AND taken_at IS NOT NULL) OR
      (status != 'taken')
    ),
  CONSTRAINT ck_medication_logs_skip_reason
    CHECK (
      (status = 'skipped' AND skip_reason IS NOT NULL) OR
      (status != 'skipped')
    )
);
```

**Constraints:**

| Type | Name | Definition | Rationale |
|---|---|---|---|
| CHECK | `ck_medication_logs_taken_at` | `taken` status requires `taken_at` | Ensures data completeness |
| CHECK | `ck_medication_logs_skip_reason` | `skipped` status requires `skip_reason` | Forces user to provide reason |
| UNIQUE | `uq_medication_logs_idempotency` | `(medicine_id, scheduled_time)` | Prevents duplicate logs for same dose |

**Indexes:**

```sql
-- Idempotency: one log per medicine per scheduled time
CREATE UNIQUE INDEX idx_medication_logs_idempotent
  ON medication_logs (medicine_id, scheduled_time);

-- User's medication history for adherence calculation
CREATE INDEX idx_medication_logs_user_time
  ON medication_logs (user_id, scheduled_time DESC);

-- Pending logs that need reminder processing
CREATE INDEX idx_medication_logs_pending
  ON medication_logs (scheduled_time)
  WHERE status = 'pending';

-- Adherence analytics: status distribution per user
CREATE INDEX idx_medication_logs_user_status
  ON medication_logs (user_id, status);
```

### 4.8 `notifications`

Push notifications, in-app alerts, and reminders.

```sql
CREATE TABLE notifications (
  id          UUID                 PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id     UUID                 NOT NULL,
  type        notification_type    NOT NULL,
  title       VARCHAR(200)         NOT NULL,
  message     TEXT                 NOT NULL,
  status      notification_status  NOT NULL DEFAULT 'pending',
  metadata    JSONB                DEFAULT '{}',
  read_at     TIMESTAMPTZ,
  created_at  TIMESTAMPTZ          NOT NULL DEFAULT now(),

  CONSTRAINT fk_notifications_user
    FOREIGN KEY (user_id) REFERENCES users(id) ON DELETE CASCADE,
  CONSTRAINT ck_notifications_read_at
    CHECK (
      (status = 'read' AND read_at IS NOT NULL) OR
      (status != 'read')
    )
);
```

**`metadata` JSONB — contextual payload:**

```jsonc
{
  "appointment_id": "uuid-here",    // for appointment reminders
  "medicine_id": "uuid-here",       // for medication reminders
  "action_url": "/appointments/123" // deep link target
}
```

**Indexes:**

```sql
-- Unread notifications for a user (notification bell badge count)
CREATE INDEX idx_notifications_user_unread
  ON notifications (user_id, created_at DESC)
  WHERE status IN ('pending', 'sent');

-- All notifications for a user (notification drawer)
CREATE INDEX idx_notifications_user_created
  ON notifications (user_id, created_at DESC);

-- BullMQ worker: find pending notifications to deliver
CREATE INDEX idx_notifications_pending
  ON notifications (created_at)
  WHERE status = 'pending';
```

### 4.9 `caregiver_links`

Junction table linking caregiver users to patient users with permission levels.

```sql
CREATE TABLE caregiver_links (
  id             UUID                  PRIMARY KEY DEFAULT gen_random_uuid(),
  caregiver_id   UUID                  NOT NULL,
  patient_id     UUID                  NOT NULL,
  permission     caregiver_permission  NOT NULL DEFAULT 'view_only',
  status         link_status           NOT NULL DEFAULT 'pending',
  created_at     TIMESTAMPTZ           NOT NULL DEFAULT now(),
  updated_at     TIMESTAMPTZ           NOT NULL DEFAULT now(),
  deleted_at     TIMESTAMPTZ,

  CONSTRAINT fk_caregiver_links_caregiver
    FOREIGN KEY (caregiver_id) REFERENCES users(id) ON DELETE CASCADE,
  CONSTRAINT fk_caregiver_links_patient
    FOREIGN KEY (patient_id) REFERENCES users(id) ON DELETE CASCADE,
  CONSTRAINT ck_caregiver_links_not_self
    CHECK (caregiver_id != patient_id)
);
```

**Constraints:**

| Type | Name | Definition | Rationale |
|---|---|---|---|
| UNIQUE | `uq_caregiver_links_pair` | `(caregiver_id, patient_id) WHERE deleted_at IS NULL` | One active link per pair |
| CHECK | `ck_caregiver_links_not_self` | `caregiver_id != patient_id` | Cannot be your own caregiver |

**Indexes:**

```sql
-- One active link per caregiver-patient pair
CREATE UNIQUE INDEX idx_caregiver_links_pair_active
  ON caregiver_links (caregiver_id, patient_id)
  WHERE deleted_at IS NULL;

-- Find all caregivers for a patient
CREATE INDEX idx_caregiver_links_patient
  ON caregiver_links (patient_id)
  WHERE deleted_at IS NULL AND status = 'active';

-- Find all patients a caregiver manages
CREATE INDEX idx_caregiver_links_caregiver
  ON caregiver_links (caregiver_id)
  WHERE deleted_at IS NULL AND status = 'active';
```

### 4.10 `refresh_tokens`

JWT refresh token storage for secure token rotation. Supports token family tracking for reuse detection.

```sql
CREATE TABLE refresh_tokens (
  id          UUID         PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id     UUID         NOT NULL,
  token_hash  TEXT         NOT NULL,
  family      VARCHAR(36)  NOT NULL,
  is_revoked  BOOLEAN      NOT NULL DEFAULT FALSE,
  expires_at  TIMESTAMPTZ  NOT NULL,
  created_at  TIMESTAMPTZ  NOT NULL DEFAULT now(),

  CONSTRAINT fk_refresh_tokens_user
    FOREIGN KEY (user_id) REFERENCES users(id) ON DELETE CASCADE,
  CONSTRAINT uq_refresh_tokens_hash
    UNIQUE (token_hash)
);
```

**Token Rotation Design:**

1. Each refresh token belongs to a **family** (UUID string).
2. When a token is rotated, the old token is revoked and a new one is created in the same family.
3. If a revoked token is reused (**replay attack**), all tokens in that family are revoked immediately.
4. Expired tokens are cleaned up by a scheduled BullMQ job.

**Indexes:**

```sql
-- Lookup by token hash (used during token refresh)
-- Already covered by UNIQUE constraint on token_hash

-- Find all tokens in a family (for mass revocation on reuse detection)
CREATE INDEX idx_refresh_tokens_family
  ON refresh_tokens (family)
  WHERE is_revoked = FALSE;

-- User's active tokens (for "logout all devices")
CREATE INDEX idx_refresh_tokens_user_active
  ON refresh_tokens (user_id)
  WHERE is_revoked = FALSE;

-- Cleanup job: find expired tokens
CREATE INDEX idx_refresh_tokens_expired
  ON refresh_tokens (expires_at)
  WHERE is_revoked = FALSE;
```

### 4.11 `audit_logs`

Append-only table for compliance and debugging. **No updates or deletes** are ever performed on this table.

```sql
CREATE TABLE audit_logs (
  id           UUID          PRIMARY KEY DEFAULT gen_random_uuid(),
  actor_id     UUID,
  entity_type  VARCHAR(50)   NOT NULL,
  entity_id    UUID          NOT NULL,
  action       audit_action  NOT NULL,
  old_values   JSONB,
  new_values   JSONB,
  ip_address   VARCHAR(45),
  user_agent   TEXT,
  created_at   TIMESTAMPTZ   NOT NULL DEFAULT now(),

  CONSTRAINT fk_audit_logs_actor
    FOREIGN KEY (actor_id) REFERENCES users(id) ON DELETE SET NULL

```

> **Note:** `actor_id` uses `ON DELETE SET NULL` because audit records must survive user deletion. A `NULL` actor indicates a system action or a deleted user.

**Indexes:**

```sql

CREATE INDEX idx_audit_logs_entity

-- Lookup all actions by a specific user

  ON audit_logs (actor_id, created_at DESC)
  WHERE actor_id IS NOT NULL;

-- Time-range queries for compliance review
CREATE INDEX idx_audit_logs_created_at
  ON audit_logs (created_at DESC);

-- Filter by action type
CREATE INDEX idx_audit_logs_action
  ON audit_logs (action, created_at DESC);
```

## 6. Soft Delete Strategy

### Tables with Soft Deletes

| Table | `deleted_at` Column | Rationale |
|---|---|---|
| `users` | ✅ | Account deactivation; preserve for data recovery |
| `appointments` | ✅ | Historical records needed for analytics |
| `medicines` | ✅ | Prescription history must be preserved |
| `caregiver_links` | ✅ | Audit trail of past caregiver relationships |

### Tables WITHOUT Soft Deletes

| Table | Rationale |
|---|---|
| `doctor_profiles` | 1:1 with user; cascades with user |
| `medication_logs` | Append-only; never deleted (compliance) |
| `vitals` | Append-only; never deleted (medical record) |
| `notifications` | Low-value; can be hard-deleted after TTL |
| `queue_tokens` | Ephemeral daily data; can be archived/purged |
| `refresh_tokens` | Security tokens; hard-delete on expiry |
| `audit_logs` | Append-only; never deleted (compliance) |

### Query Convention

All application queries **MUST** include the soft delete filter:

```sql
-- Prisma: Applied automatically via middleware
WHERE deleted_at IS NULL
```

**Prisma middleware for automatic soft delete filtering:**

```typescript
// prisma/middleware/softDelete.ts
prisma.$use(async (params, next) => {
  const softDeleteModels = ['User', 'Appointment', 'Medicine', 'CaregiverLink'];

  if (softDeleteModels.includes(params.model ?? '')) {
    if (params.action === 'delete') {
      params.action = 'update';
      params.args.data = { deletedAt: new Date() };
    }
    if (params.action === 'findMany' || params.action === 'findFirst') {
      params.args.where = { ...params.args.where, deletedAt: null };
    }
  }

  return next(params);
});
```

## 7. Timestamp Conventions

### Standard Columns

Every table includes at minimum:

| Column | Type | Default | Description |
|---|---|---|---|
| `created_at` | `TIMESTAMPTZ` | `now()` | Row creation time; **never updated** |
| `updated_at` | `TIMESTAMPTZ` | `now()` | Last modification time; **auto-updated via trigger** |

> **Exception:** Append-only tables (`medication_logs`, `vitals`, `notifications`, `audit_logs`, `refresh_tokens`) omit `updated_at` since rows are never modified.

### Auto-Update Trigger

```sql
-- Generic trigger function for updated_at
CREATE OR REPLACE FUNCTION fn_update_timestamp()
RETURNS TRIGGER AS $$
BEGIN
  NEW.updated_at = now();
  RETURN NEW;
END;
$$ LANGUAGE plpgsql;

-- Apply to all mutable tables
CREATE TRIGGER trg_users_updated_at
  BEFORE UPDATE ON users
  FOR EACH ROW EXECUTE FUNCTION fn_update_timestamp();

CREATE TRIGGER trg_doctor_profiles_updated_at
  BEFORE UPDATE ON doctor_profiles
  FOR EACH ROW EXECUTE FUNCTION fn_update_timestamp();

CREATE TRIGGER trg_appointments_updated_at
  BEFORE UPDATE ON appointments
  FOR EACH ROW EXECUTE FUNCTION fn_update_timestamp();

CREATE TRIGGER trg_queue_tokens_updated_at
  BEFORE UPDATE ON queue_tokens
  FOR EACH ROW EXECUTE FUNCTION fn_update_timestamp();

CREATE TRIGGER trg_medicines_updated_at
  BEFORE UPDATE ON medicines
  FOR EACH ROW EXECUTE FUNCTION fn_update_timestamp();

CREATE TRIGGER trg_caregiver_links_updated_at
  BEFORE UPDATE ON caregiver_links
  FOR EACH ROW EXECUTE FUNCTION fn_update_timestamp();
```

### Timezone Policy

- All timestamps are stored as `TIMESTAMPTZ` (timestamp with time zone).
- The application server always sends and receives **ISO 8601 UTC** strings.
- The database server timezone is set to `UTC` (`SET timezone = 'UTC';`).
- Client-side rendering converts to the user's local timezone.

## 8. Data Integrity Rules & Business Constraints

### 8.1 Cross-Table Business Rules

These constraints cannot be expressed purely in DDL and must be enforced in application code (service layer):

| # | Rule | Enforcement |
|---|---|---|
| 1 | Only users with `role = 'doctor'` may have a `doctor_profiles` row | Service layer check + DB trigger (optional) |
| 2 | `appointments.patient_id` must reference a user with `role = 'patient'` | Service layer validation |
| 3 | `appointments.doctor_id` must reference a user with `role = 'doctor'` | Service layer validation |
| 4 | `caregiver_links.caregiver_id` must reference a user with `role = 'caregiver'` | Service layer validation |
| 5 | `caregiver_links.patient_id` must reference a user with `role = 'patient'` | Service layer validation |
| 6 | A patient cannot book two appointments with the same doctor in overlapping slots | Exclusion constraint or service layer |
| 7 | `token_number` must be auto-incremented per doctor per day | Calculated in a serialized transaction |
| 8 | Medication logs cannot be created for inactive medicines | Service layer check |

### 8.2 Overlap Prevention for Appointments

```sql
-- PostgreSQL exclusion constraint using btree_gist extension
CREATE EXTENSION IF NOT EXISTS btree_gist;

ALTER TABLE appointments
  ADD CONSTRAINT excl_appointments_no_overlap
  EXCLUDE USING gist (
    doctor_id WITH =,
    tstzrange(slot_start, slot_end) WITH &&
  )
  WHERE (deleted_at IS NULL AND status NOT IN ('cancelled', 'no_show'));
```

> This prevents double-booking a doctor for overlapping time ranges at the database level.

### 8.3 Token Number Generation

Token numbers must be sequential per doctor per day. This is handled in a serialized transaction:

```sql
-- Within a SERIALIZABLE transaction:
SELECT COALESCE(MAX(token_number), 0) + 1 AS next_token
FROM appointments
WHERE doctor_id = $1
  AND slot_start::DATE = $2
  AND deleted_at IS NULL
  AND status NOT IN ('cancelled', 'no_show');
```

### 8.4 Medication Log Idempotency

The unique constraint `(medicine_id, scheduled_time)` ensures that BullMQ workers creating medication log entries are idempotent. If a job is retried, the `INSERT ... ON CONFLICT DO NOTHING` pattern prevents duplicates:

```sql
INSERT INTO medication_logs (medicine_id, user_id, scheduled_time, status)
VALUES ($1, $2, $3, 'pending')
ON CONFLICT (medicine_id, scheduled_time) DO NOTHING;
```

## 9. Migration Strategy Notes

### Prisma Migrations

All schema changes go through Prisma Migrate:

```bash

npx prisma migrate dev --name <descriptive_name>

# Apply in production
npx prisma migrate deploy

# Reset (dev only!)
npx prisma migrate reset
```

### Migration Naming Convention

```
YYYYMMDDHHMMSS_<verb>_<entity>_<detail>
```

Examples:
- `20260606120000_create_users_table`
- `20260607090000_add_avatar_url_to_users`
- `20260608150000_create_vitals_partitions`

### Migration Best Practices

| Practice | Detail |
|---|---|
| **Never drop columns in production** | Add a new column, migrate data, deprecate old column, drop after verification |
| **Always add columns as nullable first** | Add `DEFAULT` or backfill, then add `NOT NULL` constraint |
| **Index creation uses `CONCURRENTLY`** | Prevents table locks on large tables: `CREATE INDEX CONCURRENTLY ...` |
| **Enum modifications** | PostgreSQL enums can only have values appended. To rename or remove: create new enum, migrate, drop old |
| **Test migrations against a staging DB** | Clone production data, apply migration, validate |
| **Seed data** | Use `prisma/seed.ts` for reference data (never for user data) |

## 10. Performance Considerations

### 10.1 Indexing Strategy Summary

| Index Type | Usage | Tables |
|---|---|---|
| **B-tree (default)** | Equality and range queries | All tables |
| **B-tree partial** | Filter on soft deletes, status | `users`, `appointments`, `medicines`, `caregiver_links` |
| **B-tree composite** | Multi-column lookups | `medication_logs(user_id, scheduled_time)`, `vitals(user_id, vital_type, recorded_at)` |
| **GIN** | JSONB containment queries | `doctor_profiles(availability)`, `medicines(timings)` |
| **GiST** | Range overlap exclusion | `appointments` (slot overlap prevention) |

### 10.2 Partitioning Strategy

#### `vitals` — Range Partitioning by `recorded_at`

Vitals is the highest-volume table (multiple readings per patient per day). Monthly range partitioning provides:

- **Faster queries**: Dashboard queries typically span 7–90 days; partition pruning skips irrelevant months
- **Efficient archival**: Old partitions can be detached and moved to cold storage
- **Parallel index builds**: Each partition has its own indexes

```sql
-- Convert vitals to a partitioned table
CREATE TABLE vitals (
  id           UUID         NOT NULL DEFAULT gen_random_uuid(),
  user_id      UUID         NOT NULL,
  vital_type   vital_type   NOT NULL,
  values       JSONB        NOT NULL,
  notes        TEXT,
  recorded_at  TIMESTAMPTZ  NOT NULL DEFAULT now(),
  created_at   TIMESTAMPTZ  NOT NULL DEFAULT now(),

  CONSTRAINT fk_vitals_user FOREIGN KEY (user_id) REFERENCES users(id) ON DELETE CASCADE
) PARTITION BY RANGE (recorded_at);

-- Create monthly partitions (automate via cron or pg_partman)
CREATE TABLE vitals_2026_01 PARTITION OF vitals
  FOR VALUES FROM ('2026-01-01') TO ('2026-02-01');
CREATE TABLE vitals_2026_02 PARTITION OF vitals
  FOR VALUES FROM ('2026-02-01') TO ('2026-03-01');
-- ... generate partitions programmatically
```

#### `medication_logs` — Range Partitioning by `scheduled_time`

Similar volume characteristics to vitals. Partition monthly:

```sql
CREATE TABLE medication_logs (
  -- columns as defined above --
) PARTITION BY RANGE (scheduled_time);
```

#### `audit_logs` — Range Partitioning by `created_at`

Audit logs grow indefinitely. Monthly partitioning enables:

- Regulatory retention policies (keep 2 years, archive older)
- Efficient compliance queries within time windows

```sql
CREATE TABLE audit_logs (
  -- columns as defined above --
) PARTITION BY RANGE (created_at);
```

#### `notifications` — Candidate for Partitioning

If notification volume grows significantly, partition by `created_at`. For MVP, a standard table with a periodic purge job is sufficient.

### 10.3 Query Performance Patterns

| Query Pattern | Optimization |
|---|---|
| Patient dashboard (vitals chart) | Composite index `(user_id, vital_type, recorded_at DESC)` + partition pruning |
| Medication adherence % | Partial index on `status` + aggregate query with date range filter |
| Doctor's daily schedule | Composite index `(doctor_id, slot_start)` |
| Live queue position | Index `(doctor_id, queue_date, token_number)` + Socket.IO cache in Redis |
| Unread notification count | Partial index on `status IN ('pending', 'sent')` |
| Admin user search | Index on `(email)` with `pg_trgm` extension for LIKE queries (future) |

### 10.4 Connection Pooling

```
PostgreSQL ← PgBouncer (transaction mode) ← Prisma Client
```

- **PgBouncer** in transaction-pooling mode to handle connection bursts from BullMQ workers and API servers.
- Prisma's connection pool size: `DATABASE_URL="...?connection_limit=10"`.
- For production: one PgBouncer instance with `max_client_conn=200`, `default_pool_size=20`.

### 10.5 Caching Layer (Redis)

| Data | Cache Strategy | TTL |
|---|---|---|
| Live queue state | Write-through (update cache on every token change) | 24 hours (auto-expire end of day) |
| Doctor availability | Cache-aside (fetch from DB on miss) | 5 minutes |
| User session data | Write-through | Matches JWT expiry |
| Dashboard analytics | Precomputed nightly by BullMQ job | Until next computation |
| Notification badge count | Write-through (increment/decrement on changes) | 1 hour |

| Table | Rows/Month | Row Size (avg) | Monthly Growth |
|---|---|---|---|
| `vitals` | ~60,000 | ~200 bytes | ~12 MB |
| `medication_logs` | ~90,000 | ~150 bytes | ~13.5 MB |
| `notifications` | ~30,000 | ~300 bytes | ~9 MB |
